第1回 卒論プロジェクトを開始する

スクリプト

1 この回のねらい

卒論の分析を始める前に、作業の土台を整えます。この回では次のことを行います。

  • 授業で使うアカウントとツールを確認する（GitHub、VS Code、uv など）
  • プロジェクトフォルダをテンプレートから作り、Python 環境をセットアップする
  • 生データの置き場所を .env で管理する仕組みを理解する
  • Penn World Table のデータを使ってスクリプトを1本動かす
  • GitHub Copilot のセットアップと使い方を確認する

2 アカウント

授業で使うサービスのアカウントを作っておいてください。

必須

  • GitHub — バージョン管理とコードの共有に使います。まだ持っていない場合は作成してください。
    • GitHub Student Developer Pack — 学校のメールアドレスと学生証の両方で申請します。承認されると GitHub Copilot Student が無料で使えるようになります。承認には数日かかる場合があります。

あると便利

  • Google — AI ツール（Gemini）や各種サービスで使います。多くの人はすでに持っています。
  • Kaggle — データセットの公開・検索ができるプラットフォームです。コンペに参加しなくても、公開データを卒論の題材探しや練習に使えます。
  • AtCoder — 競技プログラミングのサービスです。Python の練習問題として活用できます。
  • Paiza — 日本語のプログラミング学習サービスです。スキルチェックやドリルで基礎を固めるのに使えます。

3 環境

この授業で使用する各種ツールのインストールにはパッケージマネージャーを使います。パッケージマネージャーとは、ソフトウェアのインストール・アップグレード・削除をコマンド1つで管理できるツールです。この授業では以下のものを推奨します。

パッケージマネージャー:

  • (Mac) Homebrew
    • 公式サイトの手順に従ってインストールしてください。Homebrew のインストール時に Git も一緒に入ります。
  • (Windows) winget
    • winget は Windows に標準で入っています。

使用ツールと役割:

  • Git — バージョン管理。変更履歴を残し、いつでも以前の状態に戻せる
  • VS Code — コードエディタ
  • GitHub CLI (gh) — GitHub リポジトリの作成・操作をターミナルから行う
  • GitHub Desktop — commit・push などの日常操作を GUI で行う
  • uv — Python のバージョン管理と依存パッケージの管理

表 1 を参考にインストールしてください。

ノート

uv の公式ドキュメントにはスクリプトによるインストール方法も載っています。この授業では他のツールと統一して winget を使います。

ノート

gh は初回に gh auth login で認証が必要です。授業中に一度だけ行えば大丈夫です。

アップデート

ソフトウェアのアップグレードは以下のコマンドで行います。

4 プロジェクトの構成

卒論を構成するファイル（コード・データ・図表・本文）は、なるべく1つのフォルダにまとめることを強く推奨します（生データは例外、後述）。また、プロジェクトごとにフォルダ構成を変えると、どこに何があるかを毎回考え直す手間が生まれます。大体いつも同じ構成にしておくことで、余計な思考を省けます。

4.1 最小フォルダ構成

📁 my-thesis-project/
├─ 📄 thesis.docx      # 卒論本文
├─ 📁 scripts/         # Python スクリプト（加工コードも含む）
├─ 📁 data/            # 加工済みデータ・出所メモ
├─ 📁 output/          # 図表出力
├─ 📄 .env　　　　　     # 環境変数
├─ 📄 .gitignore
└─ 📄 README.md

data/ には分析に使うデータファイルを置きます。生データを加工して作った場合は、その変換スクリプトを scripts/ に置きます。生データ自体は data/ に置かず別管理にします（後述）。

これは最小構成なので、必要に応じてフォルダを追加してかまいません。たとえば、PowerPoint で作った図や手書きのメモを figures/ にまとめる、といった使い方が考えられます。

重要データの来歴を残す

data/ のファイルがどこから来たかを後から追えるようにしておくことが重要です。

  • スクリプトで変換した場合 — 変換コードを scripts/ に残す
  • 手作業で変換した場合（Excel を手動で整形するなど）— 作業手順を data/ 内のメモに残す
  • 自分で入力したデータ — 生データと分析用データが同一なので変換手順は不要。入力方法や変数の定義をメモしておく

スクリプトで変換する場合の構成は、Cookie Cutter Data Science の考え方を参考にしています。機械学習などより本格的なプロジェクトに取り組む場合は、そちらの構成も選択肢になります。

4.2 GitHub テンプレートから始める

上の最小構成をそのまま使えるよう GitHub テンプレート を用意しました。コマンド操作で、上記の構成を持つリポジトリを作ることができます。

まず、プロジェクトを置きたいフォルダに移動してからコマンドを実行します。

cd ~/repos   # 好きな場所に変えてください
gh repo create my-thesis-project \
  --private \
  --template kjst-edu/thesis-starter \
  --clone

my-thesis-project の部分はリポジトリ名になります。自分の GitHub アカウント内で重複しない名前にしてください（例: urban-wage-thesis）。

このコマンド1つで、次の構成を持つフォルダが手元にできます。

📁 my-thesis-project/
├─ 📄 thesis.docx          # 卒論本文テンプレート
├─ 📄 report.docx          # レポート本文テンプレート
├─ 📁 scripts/             # Python スクリプト
├─ 📁 data/                # 加工済みデータ・出所メモ
├─ 📁 output/              # 図表出力
├─ 📄 .env.example         # 環境変数の雛形
├─ 📄 .gitignore
└─ 📄 README.md

thesis.docx と report.docx は使う方だけ残して、もう片方は削除してかまいません。

警告クラウド同期フォルダに置かない

Mac の 書類 は iCloud Drive、Windows の ドキュメント は OneDrive で同期されている場合があります。クラウド同期フォルダ内に Git リポジトリを置くと、ファイルのロックや競合が起きることがあります。~/repos や C:\repos のように同期の対象外となるフォルダを使うことをおすすめします。

クローンが完了したら、プロジェクトフォルダに移動してエディタを開きます。

cd my-thesis-project
code .      # VS Code を現在のフォルダで開く
github      # GitHub Desktop を開く

4.3 Python 環境

分析の再現性を確保するために、プロジェクトごとに専用の Python 環境（.venv）を用意します。「このプロジェクトではどのパッケージの何バージョンを使ったか」が pyproject.toml に記録されるため、後から同じ環境を再構築できます。

テンプレートをクローンした場合は pyproject.toml がすでに入っているので、プロジェクトフォルダで uv sync を実行するだけで環境が整います。

uv sync

ゼロから始める場合は uv init で pyproject.toml を生成してから uv sync を実行します。

よく使うコマンドは次の通りです。

やりたいこと コマンド
環境をセットアップ uv sync
スクリプトを実行 uv run scripts/...
パッケージを追加 uv add パッケージ名

Jupyter を使いたい場合は、必要になった時点で追加すれば大丈夫です。

uv add jupyter ipykernel

4.4 バージョン管理

GitHub を使う理由は主に2つあります。

  • 変更を取り消しやすい。 分析を進めるうちに前の状態に戻したくなることがあります。commit しておけば、いつでも以前の状態を確認・復元できます。
  • AI を使うときに安全。 AI にコードを生成させると、意図しない変更が入ることがあります。commit しておけば変更前後の差分を確認でき、問題があれば戻せます。

日常の commit・push には GitHub Desktop を使ってかまいません。ここで大事なのは、ツールの使い方を完璧にすることではなく、「作業の区切りごとに commit する」習慣をつけることです。

5 生データの置き場所を分ける

卒論で扱うデータには、公開データもあれば、利用規約がある個票データもあります。いずれの場合も、生データをこのフォルダの外に置ける仕組みにしておくと安全です。

この授業では、生データの保存先を .env というファイルに書きます。.env は commit しません。共有するのは .env.example（雛形）だけにします。

この方法には次の利点があります。

  • 生データの保存先が人によって違ってもコードを変えなくてよい
  • 絶対パスをスクリプトに直書きしなくてよい
  • 規約上アップロードできないデータを誤って GitHub に上げるリスクを減らせる

.env のほかに、生データそのものと利用規約上アップロードできないデータも commit しません。

5.1 手順

1. 生データ用フォルダを作る

プロジェクトフォルダの外に、生データを置くフォルダを作ります。場所と名前は自由です。

  • Mac
  • Windows
mkdir -p ~/Data/my-thesis-raw
mkdir C:\Data\my-thesis-raw

2. .env.example を .env にコピーする

プロジェクトフォルダのルートで実行します。

  • Mac
  • Windows
cp .env.example .env
copy .env.example .env

3. .env にパスを書く

VS Code で .env を開き、RAW_DATA_DIR= の右側に手順 1 で作ったフォルダのパスを書きます。

  • Mac
  • Windows
RAW_DATA_DIR=/Users/yourname/Data/my-thesis-raw
RAW_DATA_DIR=C:\Data\my-thesis-raw

.env は .gitignore に含まれているため、commit されません。

ノート.gitignore とは

.gitignore は「Git の管理対象から除外するファイル・フォルダ」を列挙したファイルです。ここに書かれたファイルは git status に表示されず、誤って commit することを防げます。テンプレートには .env や __pycache__/ などがあらかじめ記載されています。

6 スクリプトを1本動かしてみる

Penn World Table（PWT）は、各国の GDP・労働・資本などを長期にわたって収録したマクロ経済データセットです。整形済みで欠損も少なく、分析の練習に向いています。ここでは PWT を使って、実際にスクリプトを動かしてみましょう。

実行前に、データとスクリプトの2つを準備します。

1. PWT データをダウンロードする

Penn World Table の配布ページ から Stata 形式（pwt110.dta）をダウンロードし、先ほど作った生データフォルダ（RAW_DATA_DIR）に置きます。

2. スクリプトをダウンロードする

GitHub リポジトリ から explore_pwt.py をダウンロードし、プロジェクトの scripts/session_01/ フォルダに置きます。

3. スクリプトを実行する

スクリプトはターミナルから uv run で実行します。VS Code のエディタ右上に表示される再生ボタン（▶）は、uv で管理している Python 環境を使わないため、ここでは使いません。

uv run scripts/session_01/01-prepare_pwt.py
uv run scripts/session_01/02-plot_pwt.py

01-prepare_pwt.py では次のことを行っています。

  • PWT（pwt110.dta）を読む
  • 対象国・期間を絞り込む
  • 一人あたり実質 GDP を計算する
  • 加工済みデータを data/ に保存する

02-plot_pwt.py では次のことを行っています。

  • 加工済みデータを data/ から読む
  • 折れ線グラフを作成して output/ に保存する

分析結果の解釈よりも、「スクリプトを走らせると data/ と output/ にファイルができる」流れをつかむことを優先します。

7 AI ツール

7.1 GitHub Copilot

GitHub Copilot（無料プラン）は最近の VS Code に最初から組み込まれています。GitHub アカウントでサインインするだけで使えます。GitHub Student Developer Pack の承認後は Copilot Student に切り替わります。Gemini、Claude など他のサービスのサブスクリプションも拡張機能経由で使えますが、ここでは割愛します。

サインインは VS Code 左下のアカウントアイコン（人のシルエット）をクリックし、「GitHub でサインイン」を選んでください。ブラウザで認証すると完了します。

VS Code に「CHAT」タブが表示されていますが、これは GitHub Copilot Chat のパネルです。「このエラーの意味は？」のように自然言語でコードについて質問できます（公式ドキュメント）。よく使うショートカットを覚えておくと便利です。

Copilot Chat のショートカット
機能 Mac Windows
チャットを開く Ctrl+Cmd+I Ctrl+Alt+I
インラインチャット（エディタ内） Cmd+I Ctrl+I
コマンドパレット Cmd+Shift+P Ctrl+Shift+P

7.2 AI との付き合い方

重要AI に制御されない

AI が生成したコードをそのまま実行するだけでは、分析の中身を自分で説明できなくなります。

  • AI の出力を読んで理解してから実行する
  • 分析の目的と手順は自分で判断する
  • commit の前後で差分を確認する

卒論の分析の責任は自分にあります。AI はあくまでツールです。

テンプレートに含まれる AGENTS.md は、AI ツールに「このプロジェクトのルール」を伝えるファイルです。「コードは scripts/ に置く」「.env は commit しない」といった指示が書かれており、AI が生成したコードでもこのルールが守られるよう機能します。自分のプロジェクトに合わせて書き足すことができます。

7.3 AI に相談するときの最小ルール

  • 生データや利用規約があるデータやAPIキーを外部 AI に送らない（規約が不明なら送らない）
  • AI に分析の判断をゆだねない — 何を調べるか・どう解釈するかは自分で決める
  • AI の出力は読んで理解してから使う

疑似データを使いたい場合は、実データの列構造だけ残して値をランダムに置き換える方法があります。
