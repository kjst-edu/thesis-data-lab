# PROJECT

## この文書の役割

この文書は、このリポジトリで何を作ろうとしているのか、そのために何を優先するのかを整理するためのものです。

細かな作業ルールや必須事項は `AGENTS.md` に書き、教材の目的や設計思想はこの文書を参照する前提にします。

## このプロジェクトのゴール

このリポジトリのゴールは、卒論準備中の学部4年生向けに、Python によるデータ分析・計量経済学の短時間教材を整備することです。

想定する教材は、1回30分程度のショートセッションを積み重ねる形式で、各手法を深く教えることよりも、卒論で最低限必要になる分析作法を身につけさせることを重視します。

## 誰のための教材か

- 卒論準備中の学部4年生
- 重回帰分析と DiD は既習
- 固定効果モデル、時系列、機械学習は未習の学生を含む
- 卒論で実際にデータを扱い始める段階の学生

## 教材で身につけてほしいこと

- データの取得元を正確に記録する
- 元データから分析用データを作るコードを残す
- 記述統計量を確認してから推定に進む
- 先に図を描き、関係の向きや外れ値を考える
- 推定結果を機械的に読むのではなく、事前の見立てと照らして考える
- AI エージェントを使う場合の安全な作法を理解する

## 範囲の取り方

- 「広く浅く」よりも「共通部分を徹底する」を優先する
- 各手法は入口を扱い、深い理論説明は自習資料に回す
- 授業ノートは先生が加筆しやすいことを優先する
- 学生向けの実行例は、Jupyter 依存を前提にしない
- マクロのシミュレーションも数回含めてよいが、全体の軸は分析作法に置く

## コンテンツ設計の基本方針

### ノートとコードを分ける

- `notes/` は Quarto ベースの講義ノート
- `scripts/` は学生向けの実行用コード
- 講義で話す内容と、学生が実行して確かめる内容を混同しない

### 1回30分で完結する密度にする

- 1回ごとに到達点を絞る
- 理論を詰め込みすぎない
- 授業で話すことと自習に回すことを分けやすくする

### 欠席者が自習できるようにする

ゼミ生は就職活動などで欠席することがある。授業に出られなかった学生が後から読んで追いつけるよう、手順や参考リンク・文献をその場で示す。ノートと参考資料を合わせれば理解できる状態を目指す。

### 実務上の注意を優先する

- データ管理
- 図表作成
- 変数確認
- 記録の残し方
- 再現可能性
- AI 利用時の注意

## AI 利用に関する考え方

- バージョン管理は必須と考える
- 生データや制約付きデータは外部サービスに送らない
- 規約が不明な場合は送らない
- 外部 AI に相談する場合は、列名、型情報、疑似データ、公開データ、エラーメッセージ、自作コードなどに限定する
- 生データを repo 外に置き、加工済みデータのみを管理する構成も積極的に認める

## Web API と secrets 管理で守ること

- バージョン管理に API secrets を入れない
- API key や token をコードに直書きしない
- `.env` やローカル設定ファイルは Git に入れない
- 共有するのは `env.example` のような雛形だけにする
- 規約やデータ利用条件が不明な API は、取得前に確認する
- 取得したデータの出所、取得日、エンドポイント、利用条件を記録する

## 変更に強い進め方

- 大きな枠組みを先に固定しすぎない
- まずは第1回と第2回を書き始めやすい最小構成を優先する
- 方針変更があっても差し替えやすいよう、理念と実務ルールを分けて管理する
- 過剰な抽象化やフレームワーク導入は避ける

## 今後の判断基準

構成に迷ったときは、次の順で優先します。

1. 卒論に必要な共通作法を教えやすいか
2. 30分の授業で扱いやすいか
3. 人間が後から加筆・修正しやすいか
4. AI を使っても安全に保守しやすいか
5. 単純で壊れにくいか
